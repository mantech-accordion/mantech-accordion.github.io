## 파이프라인

파이프라인은 이미지 빌드 및 배포를 위해 한 개 이상의 태스크를 가지고, 태스크의 연관 관계(디펜던시)를 관리합니다. <br>파이프라인은 지속적으로 수행이 가능하며 수행에 대한 이력을 관리할 수 있습니다.

![pipeline-list.png](/assets/images/v3/application/pipeline/pipeline-list.png)

파이프라인 목록 중 하나를 선택한 뒤 상단의 `파이프라인` 탭을 선택하면 파이프라인을 구성하고 있는 태스크 정보를 그래프로 확인할 수 있습니다. 그래프의 노드는 태스크를 의미하며, 노드를 선택하면 해당 태스크에 대한 정보를 조회할 수 있습니다.

![build-pipeline.png](/assets/images/v3/application/pipeline/build-pipeline.png)


상단의 `이력` 탭에서는 파이프라인을 실행한 빌드 이력을 확인할 수 있습니다. 이때 개별 빌드에 대한 상태 정보가 아이콘으로 표시되는데 각각이 의미하는 바는 다음과 같습니다.

| 이미지|  태스크 상태  |
|:----|:-------|
| ![build-status-not-builted.png](/assets/images/v3/application/pipeline/build-status-not-builted.png){: width="30" }| 종료(Terminated): 사용자가 빌드를 중지한 상태 | 
| ![build-status-successed.png](/assets/images/v3/application/pipeline/build-status-successed.png){: width="30" }| 성공(Succeeded) : 태스크가 수행하고 종료코드(exit code)가 0으로 완료된 상태  | 
| ![build-status-running.png](/assets/images/v3/application/pipeline/build-status-running.png){: width="30" }| 실행 중(Running) : 태스크가 수행되고 있는 상태  | 
| ![build-status-failed.png](/assets/images/v3/application/pipeline/build-status-failed.png){: width="30" }| 실패(Failed) : 태스크가 수행되었지만 종료코드가 0이 아닌 것으로 완료된 상태 | 
| ![build-status-failed.png](/assets/images/v3/application/pipeline/build-status-failed.png){: width="30" }| 에러(Error) : 태스크가 실행되지 못하거나 비정상 종료된 상태<br><br> [발생 예시]<br> - 쿠버네티스 인프라적인 요인으로 태스크가 수행되지 않는 경우 <br> - 수행 중인 태스크가 외부 요인에 의해서 비정상 종료되는 경우 | 
| ![build-status-pending.png](/assets/images/v3/application/pipeline/build-status-pending.png){: width="30" }| 대기(Pending) : 태스크가 수행 전 대기하는 상태 |
| ![build-status-unstable.png](/assets/images/v3/application/pipeline/build-status-unstable.png){: width="30" }| 경고(Warning): 태스크를 수정 중에 잘못된 데이터를 입력하는 경우 발생 |

빌드 이력을 선택하면 개별 빌드에 대한 상세 정보를 확인할 수 있습니다.

![build-history-detail.png](/assets/images/v3/application/pipeline/build-history-detail.png)

개별 빌드는 하나 이상의 태스크로 구성되며 태스크를 선택하면 해당 태스크에서 발생한 로그를 확인할 수 있습니다.

- 실패한 태스크의 경우 실패한 원인을 확인할 수 있습니다.

    ![build-history-log-failed.png](/assets/images/v3/application/pipeline/build-history-log-failed.png)

- 태스크에는 승인 태스크도 있습니다. 승인 태스크를 선택했을 때 화면 아래에 로그 정보가 아닌 승인 목록이 나타나고, 로그인 한 사용자가 승인자 목록에 포함되어 있을 경우 승인 또는 거절이 가능합니다.

    ![build-approval.png](/assets/images/v3/application/pipeline/build-approval.png)

    승인의 경우 모든 승인자가 승인해야 다음 태스크로 넘어가며, 한 명의 승인자라도 거절하면 파이프라인은 중지됩니다.

    - 승인 성공
        ![build-approval-allow.png](/assets/images/v3/application/pipeline/build-approval-allow.png)
    - 승인 실패
        ![build-approval-deny.png](/assets/images/v3/application/pipeline/build-approval-deny.png)

### 파이프라인 생성

파이프라인 목록의 우측 상단의 `+ 생성` 버튼을 선택하면 나타나는 화면에서 파이프라인 정보를 입력하여 생성할 수 있습니다.  생성 시에는 `FORM` 또는 `YAML`로 입력할 수 있습니다.

![build-create.png](/assets/images/v3/application/pipeline/build-create.png)

입력하는 정보는 다음과 같습니다.

| 항목        |  설명  |
|:------------|:-------|
| 파이프라인 이름 | 생성할 파이프라인 이름 |
| 파이프라인 템플릿 선택 | 파이프라인을 구성하는 태스크에 대한 정보를 템플릿으로 선택 |
| 파이프라인 미리보기 | 파이프라인을 구성하고 있는 태스크 정보를 그래프로 확인 |
| 파이프라인 요약 | 파이프라인에 대한 한줄 요약 (파이프라인 목록에 표시) |
| 파이프라인 설명 | 파이프라인에 대한 설명을 마크다운으로 작성 |

파이프라인 생성 시 파이프라인 템플릿에서 태스크에 대한 구성 정보를 불러와 설정할 수 있습니다. 
템플릿을 수정해서 사용하고 싶은 경우 파이프라인 생성 후 수정 화면에서 변경 사항을 반영합니다.

![build-select-pipelinetemplate.png](/assets/images/v3/application/pipeline/build-select-pipelinetemplate.png)

- 파이프라인 생성 시 빈 템플릿을 선택하면 태스크가 없는 파이프라인을 생성할 수 있습니다. 이 경우 처음부터 파이프라인을 모두 작성할 수 있습니다.

### 파이프라인 수정

파이프라인을 생성할 때 태스크에 대한 정보는 파이프라인 템플릿에서 가져와 구성합니다. 파이프라인의 태스크에 대한 정보를 변경해야 하는 경우 변경할 파이프라인을 찾아 파이프라인 탭의 `수정` 버튼을 선택합니다.

![pipeline-update.png](/assets/images/v3/application/pipeline/pipeline-update.png)

- 태스크가 없는 파이프라인의 경우에는 수정하여, 빈 태스크의 `+` 버튼을 선택하면 태스크 구성을 할 수 있습니다.
    ![build-create-task.png](/assets/images/v3/application/pipeline/build-create-task.png)

태스크에 대한 스펙을 작성할때에는 `태스크 이름`과 `태스크 연관관계`를 설정하고 상세정보는 태스크 템플릿을 기반으로 작성합니다.

태스크의 이름은 쿠버네티스 이름 정책에 맞춰 작성하고 태스크의 연관관계는 해당 태스크을 수행하는 조건에 대한 정보를 입력합니다. 이는 `이름.상태` 의 형식으로 입력할 수 있습니다. 상태에 입력할 수 있는 값은 다음과 같습니다.

| 상태        |  설명  |
|:------------|:-------|
| Succeeded | 성공 |
| Failed | 실패 | 
| Skipped | 생략 | 
| Error | 에러 | 

연관관계 작성 방법은 다음과 같습니다.

- 태스크 연결
    - 태스크 연관관계의 입력창을 선택한 후 태스크 설정<br>
        ![build-depends-edit1.png](/assets/images/v3/application/pipeline/build-depends-edit1.png)
- 태스크 상태, 조건 추가
    - 순차적으로 태스크의 상태와 조건을 선택<br>
        ![build-depends-edit2.png](/assets/images/v3/application/pipeline/build-depends-edit2.png)
- 태스크 삭제, 수정
    - 수정이 필요한 태스크, 상태를 클릭하여 내용 변경<br>
        ![build-depends-edit3.png](/assets/images/v3/application/pipeline/build-depends-edit3.png)

[연관관계 작성 예시]<br>
- 조건
    - parent 태스크 후 child 와 child2 태스크를 수행한다.
    - child 태스크와 child2 태스크가 둘 다 성공하면 child3 태스크를 수행한다.
- 설정<br>
    ![build-depends.png](/assets/images/v3/application/pipeline/build-depends.png)
    ![build-depends2.png](/assets/images/v3/application/pipeline/build-depends2.png)
- 주의 : 파이프라인의 작업 공간에 대한 프로비저닝 설정 구성 시, 접근 모드를 `ReadWriteOnce`로 구성할 경우, 위 예시처럼 child 와 child2 태스크가 같이 동시에 시작하는 파이프라인을 구성하면 쓰기가 실패할 수 있기 때문에 `ReadWriteMany`로 생성해야 합니다.

**태스크 템플릿**

태스크를 작성하기 위해서는 태스크 템플릿에서 필요한 태스크를 불러와 파라미터를 설정하는 과정을 거칩니다.

![build-import-tasktemplate.png](/assets/images/v3/application/pipeline/build-import-tasktemplate.png)

태스크 템플릿의 종류는 다음과 같습니다.

| 종류        |  설명  |
|:------------|:-------|
| 일반 | 일반 유형 |
| VCS | Version Control System 유형 (e.g. git, svn), 만약 git 유형으로 AWS CodeCommit ssh 인증을 사용할 경우, `kubernetes.io/ssh-auth` 타입 시크릿에 `data.ssh-login-name` 에 ssh에 사용되는 `login_name` 값을 반드시 기술해야 한다. | 
| 빌드 | 소스빌드 유형 (e.g. maven, ant, gradle) | 
| 이미지 | 컨테이너 이미지를 만드는 유형 (e.g. kaniko) | 
| 승인 | 승인 태스크로 `spec.tasks[].approvers` 에 대해 반드시 기술하지만 container를 기술하지는 않음 |
| 아티팩트 | 파이프라인마다 설정되어 있는 S3저장소에 업로드한 데이터를 조회<br>`spec.tasks[].buckets` 을 반드시 기술하지만 container를 기술하지는 않음 |

파라미터까지 설정이 완료되면 `저장` 버튼을 선택해 작성을 완료합니다.

- 파이프라인 수정 시 쿠버네티스 리소스를 참조하는 방법은 다음과 같습니다.
    - 현재 사용자 권한에 따른 쿠버네티스 리소스 목록을 선택
    - 환경변수를 입력<br>
        ![pipeline-secret.png](/assets/images/v3/application/pipeline/pipeline-secret.png)
    - 파이프라인 수정 시 쿠버네티스 리소스를 생성할 수 있는 경우는 `Secret`, `ConfigMap`, `PersistentVolumeClaim` 3가지 경우가 있습니다. 생성 화면은 각 메뉴별 생성 화면과 동일합니다.

### 파이프라인 실행 미리보기

이력 탭에서 `실행하기` 버튼을 클릭하면 파이프라인 미리보기 페이지로 이동할 수 있습니다.

![pipeline-run.png](/assets/images/v3/application/pipeline/pipeline-run.png)
![pipeline-run-diff.png](/assets/images/v3/application/pipeline/pipeline-run-diff.png)

성공한 가장 최신의 빌드를 `현재 버전` 으로서 비교 기준이 되며, 빌드를 통해 반영될 파이프라인 변경 사항을 `미리보기 버전` 으로서 비교가 가능합니다.<br>
우측 하단의 `실행하기` 버튼으로 파이프라인을 실행시킬 수 있습니다.

---

### 파이프라인 실행과 중지

`이력` 탭에서 `실행하기` 버튼을 선택하면 파이프라인의 빌드가 실행되는 모습을 확인할 수 있습니다.

![pipeline-stop.png](/assets/images/v3/application/pipeline/pipeline-stop.png)

상단의 `중지` 버튼 또는 우측의 멈춤 아이콘 버튼을 선택하면 진행 중인 파이프라인의 빌드를 중지시킬 수 있습니다.

---

### 파이프라인 재시작과 미리보기

빌드 내역 중 특정 빌드로 다시 배포하고 싶은 경우 해당 빌드 내역의 우측 버튼을 이용해 다시 배포도 가능합니다. 이 경우 다시 실행하려는 빌드가 성공으로 완료되었을 때만 가능합니다.<br>
`재시작` 버튼을 눌러서 재시작에 대한 파이프라인 미리보기 페이지로 이동할 수 있습니다.

![pipeline-restart-diff.png](/assets/images/v3/application/pipeline/pipeline-restart-diff.png)

성공한 가장 최신의 빌드를 `현재 버전` 으로서 비교 기준이 되며, 재시작을 통해 반영될 파이프라인 변경 사항을 `재시작 버전` 으로서 비교가 가능합니다.
우측 하단의 `재시작` 버튼으로 파이프라인을 재시작 시킵니다.

---

### 파이프라인 버전별 비교하기

성공한 특정 빌드 버전들에 한하여 서로 변경 사항을 비교 가능합니다.

![pipeline-diff.png](/assets/images/v3/application/pipeline/pipeline-diff.png)

리스트에서 선택한 특정 버전을 기준으로 다른 성공한 버전들과 비교할 수 있습니다.
